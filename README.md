# LLM Training Estimator

LLM Training Estimator is a unified toolkit for planning large-model training runs.
It combines memory estimation, training-time estimation, rental-cost calculation,
hardware comparison, and optional LLM-generated analysis in one workflow.

Instead of switching between multiple calculators and manually reconciling their
assumptions, you provide one structured YAML configuration and receive a normalized
result that is easier to review, compare, and share.

## Features

- Unified YAML schema for model, training, parallelism, DeepSpeed, hardware, and cost inputs
- Memory estimation per GPU, utilization, fit / OOM detection, and component breakdown
- Training-time estimation, GPU-hours, and efficiency-aware runtime modeling
- Rental-cost estimation based on hardware price profiles
- Risk classification and coverage warnings for approximate or unsupported paths
- Web UI with typed validation, English / Chinese interface switching, and progress display
- Multi-hardware comparison for training time and cost across GPU presets
- Optional Markdown summary generated by an LLM provider
- CLI, JSON, table, and Markdown output modes

## What the Pipeline Produces

For each run, the estimator can report:

- `runnable`
- `risk_level`
- average memory per GPU and memory utilization
- estimated training time
- GPU-hours
- estimated cost
- warnings and coverage notes

## Architecture

```text
YAML config
  -> schema validation
  -> memory estimation
  -> runnability and risk assessment
  -> time estimation
  -> cost calculation
  -> comparison / summary rendering
```

The memory stage runs first. If the run does not fit in memory, the result is marked
as not runnable and later stages are treated accordingly.

## Installation

### Requirements

- Python 3.10+
- Node.js 22.12+ for the web UI
- `git` with submodule support if you are cloning the repository yourself

### 1. Clone the repository

If the repository is distributed with third-party sources as submodules, clone it with:

```bash
git clone --recurse-submodules <your-repo-url>
cd llm-training-estimator
```

If you already cloned it without submodules:

```bash
git submodule update --init --recursive
```

### 2. Install the main project

```bash
python -m pip install -e .
```

### 3. Install bundled third-party backends

This project integrates two upstream analysis engines. Install them as local packages
so the estimator can import them directly:

```bash
python -m pip install -e third_party/llm-analysis
python -m pip install -e third_party/gpu-mem-calculator
```

Alternative upstream installs are also possible:

```bash
python -m pip install llm-analysis
python -m pip install gpu-mem-calculator
```

### 4. Install the web UI

```bash
cd web
npm install
cd ..
```

## Quick Start

Validate a configuration:

```bash
python -m training_estimator.cli validate configs/example_qwen_7b_zero3.yaml
```

Run an estimate in table format:

```bash
python -m training_estimator.cli estimate configs/example_qwen_7b_zero3.yaml \
  --prices configs/hardware_prices.yaml \
  --format table
```

Write JSON output:

```bash
python -m training_estimator.cli estimate configs/example_qwen_7b_zero3.yaml \
  --prices configs/hardware_prices.yaml \
  --format json \
  --output outputs/qwen_7b_full_h100_zero3.json
```

Supported formats:

- `table`
- `json`
- `markdown`

## Web UI

Start the API server:

```bash
python -m uvicorn --app-dir src training_estimator.web_api:app --reload --port 8000
```

Start the web UI in another terminal:

```bash
cd web
npm run dev
```

Then open:

```text
http://localhost:4321
```

The web interface supports:

- grouped schema editing
- type-aware input validation
- bilingual interface switching
- progress and ETA display
- immediate result rendering
- optional follow-up summary generation
- multi-GPU-preset comparison

## Hardware Comparison

The web UI can compare multiple hardware presets in one pass.
When you choose several GPU types, the system evaluates each preset with the same
model and training assumptions, then presents side-by-side differences for:

- training time
- estimated cost
- runnable state
- risk level
- average memory per GPU

This is useful for deciding whether a faster GPU is worth the extra rental cost, or
whether a cheaper cluster materially changes delivery time.

## Optional Summary Analysis

The API can request a short Markdown summary from a supported LLM provider after the
core estimate is complete. This summary is designed to highlight:

- whether the run is actionable
- memory pressure and OOM risk
- time and GPU-hour interpretation
- cost interpretation
- caveats and next steps
